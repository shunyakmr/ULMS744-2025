Navigating the Football Data Landscape

Leveraging StatsBombR and worldfootballR in 2026

The sports analytics ecosystem moves fast. Packages break, APIs change, and data providers adjust their terms of service. This workbook maps the football data landscape as it stands in 2026 and gives you a practical, working route into two of the most important open data sources available: Hudl StatsBomb and FBref/Understat via worldfootballR.

We will:

  1. access granular event-level data through StatsBomb’s open data;
  2. access aggregate season-wide statistics through worldfootballR;
  3. understand the current scraping limitations that affect day-to-day work.

The data access problem is real. Many tutorials written in previous years no longer work out of the box because upstream websites have added rate limits and bot detection. This workbook focuses on what actually works today so you can build a portfolio with confidence.

1 Learning Objectives

By the end of this workbook, you will be able to:

  • Access granular event data from StatsBomb’s free releases, including shots, passes, and pressures with (x, y) pitch coordinates.
  • Access aggregate season data from FBref/Understat through worldfootballR’s pre-built loader functions.
  • Explain the current scraping limitations that affect football data packages and adapt your workflow accordingly.

Run the following in the RStudio Console to install the packages used in this workbook.

install.packages("devtools")

# worldfootballR — football data from FBref, Transfermarkt, and Understat
install.packages("worldfootballR")

# Core tidyverse for wrangling and visualisation
install.packages("tidyverse")

Load the libraries for the session.

R script
# --- Libraries ----
library(StatsBombR)
library(worldfootballR)
library(tidyverse)

3 Hudl StatsBomb: Granular Play-by-Play Data

StatsBomb provides event-level data. Every pass, shot, dribble, pressure, and goalkeeper action is logged with (x, y) pitch coordinates. This is the gold standard for spatial football analysis: pitch maps, expected goals models, passing networks, and pressure heatmaps all start here.

3.1 What Is Available?

StatsBomb’s Open Data programme releases professional-grade match event data for selected competitions. As of 2026, the major releases include:

  • Lionel Messi’s full career across Barcelona, PSG, and Inter Miami.
  • FIFA World Cups (Men’s and Women’s).
  • Arsenal’s 2003–04 Invincibles season.
  • FA Women’s Super League seasons.
  • UEFA Euro tournaments.

StatsBomb’s free data is limited to these specific releases. You cannot pull arbitrary league seasons. For full coverage, a commercial licence is required. However, the open data is more than enough for portfolio-quality work.

3.2 Pulling StatsBomb Data

The workflow follows four steps: load the libraries, browse the available competitions, pick a competition/season, pull its match list, then pull events for a single match.

3.2.1 Load the libraries

StatsBombR provides the open data download functions. tidyverse gives us dplyr for filtering and selecting, plus the pipe (|>) for chaining steps together.

R script
library(StatsBombR)
library(tidyverse)

3.2.2 Browse the available competitions

FreeCompetitions() calls the StatsBomb open-data servers and returns a data frame with one row per competition × season combination that is currently free to use. The table has many columns; here we keep only the identifiers and names, drop duplicates with distinct(), and sort alphabetically so the list is easy to scan.

R script
# FreeCompetitions() downloads the list of open-data competitions from StatsBomb.
competitions <- FreeCompetitions()

# Inspect the competitions and seasons available.
# Use View(available_competitions) in RStudio for a spreadsheet-style view.
available_competitions <- competitions |>
  select(competition_id, competition_name, season_id, season_name) |>
  distinct() |>
  arrange(competition_name, season_name)

available_competitions

3.2.3 Select a competition and season

Once you know what is available, filter the competitions table down to the specific competition and season(s) you want to analyse. We use named variables (target_competition_name, target_seasons) so the choice is easy to change later. The if block gives a clear error message if the filter returns nothing — usually caused by a typo in the competition or season name.

R script
# Change these values if you want to use a different competition.
target_competition_name <- "La Liga"
target_seasons <- c("2020/2021")

selected_competition <- competitions |>
  filter(
    competition_name == target_competition_name,
    season_name %in% target_seasons
  )

# Stop early with a useful message if the competition/season filter is wrong.
if (nrow(selected_competition) == 0) {
  stop("No competition found. Check target_competition_name and target_seasons.")
}

3.2.4 Download the match list

FreeMatches() takes the competition/season data frame from the previous step and returns every match in those seasons, one row per match. We convert match_date to a proper Date type and sort chronologically so the earliest fixtures appear first.

R script
matches <- FreeMatches(Competitions = selected_competition) |>
  mutate(match_date = as.Date(match_date)) |>
  arrange(match_date)

# Inspect the match list in the RStudio viewer.
matches

3.2.5 Download event data for one match

get.matchFree() takes a single-row data frame (one match) and returns its full event log: every pass, shot, dribble, pressure, and so on, with (x, y) pitch coordinates where relevant. We use slice(1) here to pick the first match in the list as an example — swap in any match_id from the table above when you want to analyse a different game.

R script
# Use the first match as an example. You can replace this with any match_id
# from the table above.
selected_match <- matches |>
  slice(1)

events <- get.matchFree(selected_match)

# Event data is the main StatsBomb event-by-event table.
# Key columns: type.name, location, pass.end_location, shot.statsbomb_xg
glimpse(events)

3.2.6 Filter to shots and inspect spatial data

Event data is long and wide: one row per event, with many columns that are only populated for certain event types (e.g. shot.* columns are only filled when type.name == "Shot"). Filtering to shots is a typical first step for building shot maps or exploring xG.

R script
shots <- events |>
  filter(type.name == "Shot") |>
  select(
    player.name,
    team.name,
    location,
    shot.end_location,
    shot.statsbomb_xg,
    shot.outcome.name
  )

# Each 'location' is a list of [x, y] coordinates on a 120x80 pitch.
shots

3.3 Why This Matters for Your Portfolio

StatsBomb data lets you demonstrate that you can:

  • Parse and reshape complex nested JSON structures into tidy data frames.
  • Work with spatial coordinates to produce pitch maps and heatmaps.
  • Build analyses around professional metrics like xG (expected goals).

These are skills that football clubs, agencies, and analytics companies actively recruit for.

4 worldfootballR: FBref & Understat

Where StatsBomb gives you event-level depth, worldfootballR gives you breadth. It wraps multiple upstream sources — primarily FBref (powered by StatsBomb aggregates) and Understat — into a single R interface for season-wide statistics.

4.1 FBref: Advanced Aggregate Stats

FBref is the go-to source for detailed per-player and per-team season tables: shooting, passing, possession, defensive actions, and more. The data originates from StatsBomb but is presented in aggregate form (per-90 rates, totals, percentiles) rather than event-level.

4.2 Understat: xG-Focused Match and Shot Data

Understat provides its own expected goals (xG) model with match-level and shot-level data. It is a useful complement to FBref when you want shot maps or match-by-match xG timelines without needing the full granularity of StatsBomb events.

WarningCoverage Is Limited to Top Leagues

Both FBref and Understat focus on major European leagues — typically the Big 5 (Premier League, La Liga, Bundesliga, Serie A, Ligue 1) plus selected international tournaments. If your project targets a smaller league (e.g., Eredivisie, Liga Portugal, MLS), data availability will be patchy or nonexistent through these sources. Always check coverage before committing to a project scope.

ImportantScraping Functions Are Currently Unreliable

As of 2026, most live scraping functions in worldfootballR (e.g., fb_player_scouting_report(), fb_team_match_log_stats()) are broken or intermittently blocked by FBref’s anti-bot protections. Understat scrapers (understat_league_season_shots(), etc.) are more stable but can also fail without warning.

The reliable standard is to use load_ functions. These pull from pre-scraped, cached datasets that are updated regularly and do not trigger rate limits or blocks.

# DO NOT rely on this in 2026:
# fb_player_scouting_report("https://fbref.com/...")

# USE THIS INSTEAD:
load_fb_big5_advanced_season_stats(season_end_year = 2025)

If a scraping function works today, it may not tomorrow. Build your workflow around load_ functions for reproducibility.

4.3 FBref: Using the Loader Functions

4.3.1 1. Pull a season of advanced stats

load_fb_big5_advanced_season_stats() is a pre-built loader: instead of scraping FBref live, it pulls from a regularly updated cached dataset, which is why it is reliable. Two arguments matter:

  • season_end_year — the calendar year the season ends in. For the 2024/25 season, use 2025.
  • stat_type — which stat table to return (shooting, passing, defence, etc.). We start with "standard" (goals, assists, minutes, xG).

glimpse() gives a quick column-by-column preview, which is useful because these tables often have 40+ columns with suffixes like _Standard or _Expected that indicate which FBref section the stat came from.

R script
# Pull advanced stats for the Big 5 European leagues (2024-25 season)
big5_stats <- load_fb_big5_advanced_season_stats(
  season_end_year = 2025,
  stat_type = "standard"
)

# Inspect the structure
glimpse(big5_stats)

4.3.2 2. Filter and rank players

Once the data is in memory, the usual dplyr verbs apply. Here we narrow to Premier League forwards, keep only the columns we care about, and sort by non-penalty goals. MP_Playing is minutes-played-adjusted match count; npxG_Expected is the FBref naming convention for non-penalty expected goals.

R script
# Filter to Premier League forwards and rank by non-penalty goals
big5_stats |>
  filter(Comp == "Premier League", Pos == "FW") |>
  select(Player, Squad, MP_Playing, Gls_Standard, npxG_Expected) |>
  arrange(desc(Gls_Standard)) |>
  head(15)

4.3.3 3. Switch to a different stat type

The same loader returns completely different tables depending on stat_type. Swap the argument value to pull shooting, passing, possession, defensive, or goalkeeping stats — each comes with its own set of columns. Here we pull defensive stats and rank La Liga players by tackles.

R script
# Available stat types for load_fb_big5_advanced_season_stats():
# "standard", "shooting", "passing", "passing_types",
# "gca", "defense", "possession", "playing_time",
# "misc", "keepers", "keepers_adv"

# Example: Pull defensive stats
defensive <- load_fb_big5_advanced_season_stats(
  season_end_year = 2025,
  stat_type = "defense"
)

defensive |>
  filter(Comp == "La Liga") |>
  select(Player, Squad, Tkl_Tackles, Int, Blocks_Blocks) |>
  arrange(desc(Tkl_Tackles)) |>
  head(10)

4.4 Understat: Shot-Level xG Data

Understat’s value is its shot-level xG data with match context — useful for building xG timelines, shot maps, and team-level performance trends.

4.4.1 1. Pull match-level xG results

understat_league_match_results() returns one row per match in the chosen league season, including goals scored and Understat’s own xG values for each side. The league argument takes a short code ("EPL", "La_liga", "Bundesliga", "Serie_A", "Ligue_1", "RFPL") and season_start_year takes the year the season starts in — so the 2024/25 season is 2024.

R script
# Pull match-level xG results for a league season
# Understat covers: EPL, La Liga, Bundesliga, Serie A, Ligue 1, RFPL
epl_results <- understat_league_match_results(
  league = "EPL",
  season_start_year = 2024
)

# Each row is a match with home/away xG
epl_results |>
  select(home_team, away_team, home_goals, away_goals, home_xG, away_xG) |>
  head(10)

4.4.2 2. Pull shot-level data

understat_league_season_shots() goes one level deeper: one row per shot, with its xG value, outcome, pitch coordinates, and situation (open play, set piece, counter-attack, etc.). This is the table you want for shot maps or for rebuilding season-level xG trends from the ground up. Note the team = h_team inside select() — we rename the home-team column to team on the way through to make the output easier to read.

R script
# Pull shot-level data for a specific league season
epl_shots <- understat_league_season_shots(
  league = "EPL",
  season_start_year = 2024
)

# Each row is a single shot with xG, coordinates, and result
epl_shots |>
  select(player, team = h_team, xG, result, X, Y, situation) |>
  head(10)

Understat’s (X, Y) coordinates are normalised to a 0–1 scale (proportion of pitch length and width). To plot them on a standard pitch, multiply X by the pitch length and Y by the pitch width. This is a different coordinate system from StatsBomb’s 120×80 grid — always check the documentation before mapping.

5 Career and Portfolio Value

TipInsight: Why This Data Matters for Your Career

Even with a limited number of open seasons and the scraping challenges we have covered, these datasets are gold for employers. Here is why:

  • Complex data wrangling. StatsBomb’s nested JSON structure forces you to demonstrate real data engineering skills — unnesting lists, reshaping wide-to-long, and handling missing values. This is exactly what analysts do on the job.
  • Spatial data mastery. Pitch maps, pass networks, and pressure heatmaps built from (x, y) coordinates show that you can work with spatial data — a transferable skill that applies well beyond football.
  • Understanding of modern data engineering hurdles. Employers know that data access is messy. Showing that you can navigate broken APIs, rate limits, and caching strategies demonstrates professional maturity. A portfolio that uses load_ functions and documents why is more impressive than one that pretends scraping always works.
  • Domain storytelling. Turning raw event data into a clear tactical insight — “This team’s high press collapsed after the 60th minute” — shows you can communicate with non-technical stakeholders.

A single well-executed StatsBomb pitch map with a clear narrative will stand out more than a dozen generic bar charts. Focus on depth over breadth.

6 Conclusion: Credits and Data Ethics

Open data exists because organisations choose to share it. Respecting their terms of service keeps the ecosystem alive for everyone.

6.1 Mandatory Credits

When you publish work that uses these data sources — whether on GitHub, in a portfolio, or in a presentation — you must include proper attribution.

StatsBomb:

Data provided free of charge by StatsBomb. StatsBomb data is subject to the StatsBomb Public Data User Agreement.

FBref / Sports Reference:

Data sourced from FBref.com, provided by Sports Reference. Please review the Sports Reference data use terms.

Understat:

Data sourced from Understat.com. Understat provides xG models and shot data under its own terms of use.

Add these credits to the README.md of any GitHub repository that uses this data. It takes 30 seconds and it signals to employers that you understand data governance — a detail that separates professional analysts from hobbyists.

6.2 A Note on Responsible Use

  • Do not hammer FBref with rapid automated requests. This is why scraping functions break — overuse triggers protections that affect the entire community.
  • Cache your data locally. Pull once, save to a .csv or .rds file, and work from the local copy. This is faster, reproducible, and respectful.
  • Check terms before publishing. Some data providers restrict commercial use or redistribution. When in doubt, link to the source rather than redistributing the raw data.
