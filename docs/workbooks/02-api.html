2. API & Data Sourcing

Last week, we set up the project and connected it to GitHub. This week we move into the first core analytical task: sourcing and preparing sports data through an API workflow in R.

We will use hoopR to pull NBA basketball data. Basketball remains our common seminar example because it gives us rich public data and a clear route into visualisation and dashboard building. The workflow, however, is transferable: in another sport, the package and variables will change, but the project logic will stay the same.

We will:

  1. pull team and player data with hoopR;
  2. inspect the returned data frames;
  3. clean and summarise them into usable tables;
  4. save those tables for later weeks;
  5. commit and push the updated project.

2 A Publishing Shortcut: _publish.sh

Last week, we committed and pushed work to GitHub using three separate Terminal commands. That works, but typing the same commands every time gets repetitive. Let’s create a small shortcut script that does it all in one step.

2.1 Create the File

In the Files pane in RStudio:

  1. click New File
  2. choose Text File
  3. save it as _publish.sh in your project root (the same level as nba-analytics.Rproj)
  4. paste the following into the file and save
_publish.sh
# _publish.sh — A shortcut to save and publish your work to GitHub.
#
# Usage (in your R terminal):
#   ./_publish.sh "Your commit message here"
#
# Example:
#   ./_publish.sh "Add cleaned NBA team and player data"

git add .
git commit -m "$1"
git push -u origin main

echo "Done! Your work has been published to GitHub."

2.2 How to Use It

From now on, whenever you want to save and publish your work, open the RStudio Terminal and run:

RStudio Terminal
./_publish.sh "Your commit message here"

Replace the text inside the quotes with a short description of what you did. For example:

RStudio Terminal
./_publish.sh "Add cleaned NBA team and player data"

That single command will stage all your changes, commit them with your message, and push everything to GitHub. You will use this throughout the rest of the module.

The underscore in _publish.sh is intentional — it keeps the file near the top of your file list and signals that it is a helper script, not part of the main project output.

If you get a permission error, run chmod +x _publish.sh in your terminal just once, and this should make _publish.sh executable.

3 Why This Matters

Analysts are often judged on whether they can obtain and prepare their own data, not only on whether they can plot a spreadsheet someone else has already cleaned.

The workflow for today is:

documentation -> choose parameters -> pull -> inspect -> summarise -> save

That sequence is one of the key transferable habits from this module.

5 SportsDataverse Workflow

5.1 Reading the Documentation First

Before calling any data-sourcing function, you should read its documentation. The documentation tells you what data sources are available, what functions to call, what parameters they accept, and what kind of data they return. This saves time and avoids guesswork.

5.1.1 Start with the Package Website

Every SportsDataVerse package has a documentation website. This is the best place to start because it gives you the big picture: what data sources the package can access, what functions are available, and how to get started.

For hoopR, the getting-started guide is here: https://hoopr.sportsdataverse.org/articles/getting-started-hoopR.html

From that page, you can see the main functions the package offers (e.g. load_nba_team_box(), load_nba_player_box()), what data they return, and how to use them. Browse the package website to discover what is available before you start writing code.

5.1.2 Then Check Individual Functions in R

Once you know which function you want to use, check its detailed documentation in RStudio by typing ? followed by the function name:

R console
?load_nba_team_box

This opens the help page. The three sections to focus on are:

  • Usage — shows the function signature and its default parameter values.
  • Arguments — lists every parameter, explains what it controls, and tells you what values are valid.
  • Value — describes what the function returns (usually a data frame with specific columns).

5.1.3 What You Learn from the hoopR Docs

Reading the documentation for load_nba_team_box(), you discover:

  • It has one key parameter: seasons.
  • seasons accepts a numeric vector — you can request a single season (e.g. 2024) or multiple seasons (e.g. 2020:2024).
  • The function returns a data frame where each row is one team’s performance in one game.

Knowing this before you run the code means you can choose sensible parameter values on the first attempt.

5.1.4 Example: A Football Package

The same documentation habit applies to every sports data package. For example, worldfootballR provides football (soccer) data from sources including FBref and Transfermarkt. If you wanted to pull transfer data from Transfermarkt, you would start with the package documentation:

https://jaseziv.github.io/worldfootballR/articles/extract-transfermarkt-data.html

That page shows you which functions access Transfermarkt data, what parameters they require, and what the returned data looks like. Once you have identified the function you need, check its details in R:

R console
?tm_player_market_values

The Arguments section would tell you parameters like:

  • country_name — e.g. "England" for the English leagues
  • start_year — the year the season starts, e.g. 2023 for the 2023/24 season
  • league_url — optionally, a direct Transfermarkt league URL

So a call might look like:

R code
epl_values <- tm_player_market_values(
  country_name = "England",
  start_year = 2023
)

You do not need to memorise every parameter. You need to know where to look: start with the package website to see what is available, then use ?function_name in R to check the details. The documentation is always your first stop.

5.2 Step 1: Pull Team Box Score Data

load_nba_team_box() downloads team-level box score data from ESPN via the hoopR API wrapper. Each row in the result represents one team’s performance in one game, with columns for points, shooting percentages, rebounds, assists and more.

R code
nba_team_box <- load_nba_team_box(seasons = 2024)

saveRDS(nba_team_box, "data/raw/nba_2024_team_box.rds")
write_csv(nba_team_box, "data/raw/nba_2024_team_box.csv")

In hoopR, seasons = 2024 means the NBA season that ends in 2024, i.e. the 2023/24 season. This convention is common across SportsDataVerse packages.

5.3 Step 2: Inspect the Raw Data

Never summarise a dataset before checking what it contains.

R code
dim(nba_team_box)

dim() returns the number of rows and columns. This tells you how large the dataset is — for example, 2460 46 would mean 2,460 rows and 46 columns.

R code
names(nba_team_box)

names() lists every column name. Scan through this to find the variables you care about (team name, score, shooting stats, etc.).

R code
glimpse(nba_team_box)

glimpse() is the most informative single inspection command. It shows each column’s name, type (e.g. <chr>, <dbl>) and first few values — all in one view.

R code
head(nba_team_box)

head() shows the first six rows of the data frame, so you can see actual data values in context.

You can also inspect a data frame by clicking its name in the Environment panel (top-right in RStudio). This opens a spreadsheet-style viewer where you can scroll through rows and columns, sort by clicking column headers, and search for values. It is a quick way to get a feel for the data before writing any code.

Look for:

  • one row per team per game;
  • the main team identifier;
  • score and result columns;
  • the shooting, rebounding and possession-related variables you may want later.

5.4 Step 3: Prepare a Season-Level Team Table

We want one row per team, using regular-season games only. To get there, we need to group the game-level rows by team and then summarise each group into a single row of season averages.

The variables we use below (e.g. team_score, field_goals_made, total_rebounds) come from inspecting the data in the previous step. You will need to look at what columns are available in your data and decide which ones are useful for your summary. Use names() or the Environment viewer to check what is there.

R code
nba_regular <- nba_team_box |>
  filter(season_type == 2)

nba_team_stats <- nba_regular |>
  group_by(team_short_display_name) |>
  summarise(
    games = n(),
    wins = sum(team_winner == TRUE, na.rm = TRUE),
    losses = sum(team_winner == FALSE, na.rm = TRUE),
    avg_pts = round(mean(team_score, na.rm = TRUE), 1),
    avg_fg_pct = round(
      mean(field_goals_made / field_goals_attempted * 100, na.rm = TRUE),
      1
    ),
    avg_3pt_made = round(mean(three_point_field_goals_made, na.rm = TRUE), 1),
    avg_rebounds = round(mean(total_rebounds, na.rm = TRUE), 1),
    avg_assists = round(mean(assists, na.rm = TRUE), 1),
    avg_turnovers = round(mean(turnovers, na.rm = TRUE), 1),
    avg_steals = round(mean(steals, na.rm = TRUE), 1),
    avg_blocks = round(mean(blocks, na.rm = TRUE), 1),
    .groups = "drop"
  ) |>
  mutate(
    win_pct = round(wins / games * 100, 1)
  ) |>
  arrange(desc(win_pct))

A few things to note about the code above:

  • group_by(team_short_display_name) splits the data into separate groups, one per team. Everything inside summarise() is then calculated within each group rather than across the whole dataset.
  • summarise() collapses each group into a single row. The columns you create inside it (e.g. games = n(), avg_pts = round(mean(...))) define what the summary table contains.
  • .groups = "drop" tells R to remove the grouping structure after summarising. Without this, the data frame would remain grouped, which can cause unexpected behaviour in later operations.
  • mutate() adds or modifies columns without changing the number of rows. Here we use it to calculate win_pct from the wins and games columns that summarise() just created.
  • avg_fg_pct is calculated from field_goals_made / field_goals_attempted * 100 because the raw data provides made and attempted counts rather than a pre-calculated percentage.

Inspect the cleaned table.

R code
print(nba_team_stats, n = 30)
summary(nba_team_stats)
colSums(is.na(nba_team_stats))

5.5 Step 4: Prepare a Player Table

We also need a player-level table for later charts and dashboard views. The same group_by() |> summarise() pattern applies — this time grouping by player name and team.

Again, the column names below (e.g. points, rebounds, field_goals_made) come from inspecting the player box score data. Run names(nba_player_box) or click the data frame in the Environment panel to see what columns are available, and choose the ones that are relevant to your analysis.

R code
nba_player_box <- load_nba_player_box(seasons = 2024)

saveRDS(nba_player_box, "data/raw/nba_2024_player_box.rds")
write_csv(nba_player_box, "data/raw/nba_2024_player_box.csv")

nba_player_stats <- nba_player_box |>
  filter(season_type == 2, minutes > 0) |>
  group_by(athlete_display_name, team_short_display_name) |>
  summarise(
    games = n(),
    avg_pts = round(mean(points, na.rm = TRUE), 1),
    avg_rebounds = round(mean(rebounds, na.rm = TRUE), 1),
    avg_assists = round(mean(assists, na.rm = TRUE), 1),
    avg_fg_pct = round(
      mean(field_goals_made / field_goals_attempted * 100, na.rm = TRUE),
      1
    ),
    avg_3pt_made = round(mean(three_point_field_goals_made, na.rm = TRUE), 1),
    avg_minutes = round(mean(minutes, na.rm = TRUE), 1),
    .groups = "drop"
  ) |>
  filter(games >= 40) |>
  arrange(desc(avg_pts))
R code
head(nba_player_stats, 20)

5.6 Step 5: Save Reusable Data Files

5.6.1 What Is an .rds File?

An .rds file is R’s native format for saving a single object to disk. It preserves the R object exactly — column types, factor levels, attributes and all. When you load it back with readRDS(), you get the same object you saved.

We also save .csv versions because CSV is a universal format that can be opened in Excel, Python, or any other tool. Use .rds for your own R workflow; use .csv when sharing with others.

R code
saveRDS(nba_team_stats, "data/cleaned/nba_2024_team_stats.rds")
write_csv(nba_team_stats, "data/cleaned/nba_2024_team_stats.csv")

saveRDS(nba_player_stats, "data/cleaned/nba_2024_player_stats.rds")
write_csv(nba_player_stats, "data/cleaned/nba_2024_player_stats.csv")

Check that the files exist.

R code
list.files("data/raw")
list.files("data/cleaned")

Saving prepared data is part of the workflow, not a tidying afterthought. It means you do not need to re-run long API calls every time you want to make a chart.

6 Independent Practice: Your Own Sport

The basketball walkthrough above is the shared seminar example, but your individual assignment will ask you to source, clean and present data for a sport of your own choosing. This is a good opportunity to get a head start.

6.1 What to Do

  1. Pick a sport and a SportsDataVerse package (or another R package that provides sports data through an API). Some options:

    Sport Package Documentation
    WNBA / women’s college basketball wehoop https://wehoop.sportsdataverse.org/
    NFL (American football) nflreadr https://nflreadr.nflverse.com/
    MLB (baseball) baseballr https://billpetti.github.io/baseballr/
    Football (soccer) worldfootballR https://jaseziv.github.io/worldfootballR/
    F1 (motorsport) f1dataR https://scasanova.github.io/f1dataR/

  2. Install the package if you have not already, and load it.

  3. Read the documentation for the main data-loading function in that package. Use ?function_name in the Console or browse the package website. Identify:

    • what parameters the function requires;
    • what values are valid for each parameter;
    • what kind of data frame it returns.

  4. Pull one season of data using the function and your chosen parameters. Save the downloaded table in data/raw/ before you clean or summarise it.

  5. Inspect the result using dim(), names(), glimpse() and head().

  6. Summarise the data into a clean table with one row per team (or per player, driver, etc.), keeping the variables you find most interesting.

  7. Save both .rds and .csv versions of the cleaned table to your data/cleaned/ folder.

The individual assignment will ask you to do exactly this kind of work — source data from an API, prepare it, and build outputs from it. The more you practise now, the less you will need to figure out later. Think of this exercise as the first step of your assignment.

6.2 Suggestions for Football (worldfootballR)

If you are choosing football, the worldfootballR package gives you access to data from FBref and Understat, including advanced stats like expected goals (xG). Note that most live scraping functions are currently unreliable due to anti-bot protections — use the load_ functions (e.g., load_fb_big5_advanced_season_stats()) as the reliable standard, and be aware that coverage is generally limited to the Big 5 European leagues. For a full guide to the football data landscape, including StatsBomb event data and Understat shot-level xG, see the Football Data Landscape workbook.

Before pulling data, think about what story your dashboard will tell. The data you pull now shapes the dashboard you build later. Here are four different angles to get you started — each leads to a different kind of dashboard.

Here are my recommended resources for data visualisation:

  • Fundamentals of Data Visualization by Claus Wilke [link]

  • Financial Times Visual Vocabulary [link]

  • R Graph Gallery [link]

  • ULMS562 Chatper 4 by Shunya Kimura [link] (Python-based but has some useful videos)

6.2.1 Angle 1: League Overview — “Who Over- or Under-Performed?”

Story: Compare all teams in a league on the required measures. Which teams scored more than their xG suggested? Which defences conceded less than expected? A league-wide view is the most direct route to the assignment measures.

Dashboard idea: A league table with the required stats, scatter plots of xG vs actual goals, and bar charts ranking teams.

FBref’s standard stats table includes columns for goals, assists, xG, xAG, and progressive actions — most of what you need in one call. Inspect the result with glimpse() to see what is there.

6.2.2 Angle 2: One Team’s Season — “How Did Their Form Change?”

Story: Pick a single team and track their results across the season. Did they start strong and fade? Did a change in tactics show up in the xG numbers? This gives you a time-series narrative and works well with line charts and rolling averages.

Dashboard idea: Match-by-match results for the chosen team, with filters or a team selector. Cumulative points, goals scored/conceded over time, and xG trends.

This returns one row per match with columns for home/away teams, scores, and xG. From here, you can filter to one team and reshape — just as we did with the NBA team box scores.

6.2.3 Angle 3: Player Spotlight — “Who Stood Out?”

Story: Compare players within a league or a team. Who created the most chances? Who outperformed their xG? A player-level focus lets you tell a scouting-style story and works well with scatter plots and ranked tables.

Dashboard idea: Player ranking tables, scatter plots (e.g. xG vs actual goals, minutes vs goal contributions), with filters for team or position.

Try different stat_type values — "standard", "shooting", "passing", "defense" — to see what player-level data is available. You can pull multiple stat types and join them together.

6.2.4 Angle 4: Cross-League Comparison — “How Do Leagues Differ?”

Story: Pull the same stats for two or more leagues (e.g. WSL and NWSL, or Premier League and La Liga) and compare. Are goals more common in one league? Is xG higher? This creates a comparative narrative and works well with grouped bar charts and faceted plots.

Dashboard idea: Side-by-side league summaries, with tabs or a dropdown to switch between leagues.

worldfootballR’s live scraping functions (e.g., fb_team_match_log_stats()) can be slow and unreliable — FBref’s anti-bot protections frequently block them. Prefer load_ functions where available. When you do scrape, pull the data once, save it to .rds, and work from the saved file afterwards — exactly the workflow we practised with basketball. See the Football Data Landscape workbook for details.

7 Transfer Checkpoint

The details are basketball-specific, but the project habit is not. For another sport, the equivalent questions would be:

  • what is the raw event or match-level table?
  • what should one cleaned team-level row represent?
  • what should one cleaned player, driver, batter or club row represent?
  • which variables need to be saved for charting and dashboards later?

8 Common Issues

Problem Solution
hoopR call fails Check your internet connection and retry
Returned data frame is very wide Use glimpse() and select() to focus on relevant variables
Column names differ slightly Check names() and adapt the code
Missing values appear Use na.rm = TRUE and inspect where they occur
Package not found Run install.packages("package_name") first
Documentation page is empty Try the package website instead of ? — some packages document online only
